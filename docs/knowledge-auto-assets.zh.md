# 知识库与自动发送资料

## 两种能力的区别

### 1. 知识库

知识库用于给 AI 提供检索上下文。

处理链路：

1. 用户发消息。
2. 系统根据当前客服 Agent 绑定的知识库做检索。
3. 命中的文本片段被追加到 system prompt。
4. AI 基于这些上下文生成自然语言回复。

知识库的输出是“帮助 AI 回答”，不是直接给用户发文件。

### 2. 自动发送资料

自动发送资料是独立于知识库的客服 Agent 配置。

它的用途是：

- 用户问“给我操作手册”时，自动附上 PDF
- 用户问“发我演示视频”时，自动附上视频
- 用户问“有没有安装说明”时，自动附上对应资料链接

这条链路不会替代知识库，而是和知识库并行。
## 当前实现

在客服 Agent 配置中新增了“自动发送资料”规则。

每条规则包含：

- 规则名称
- 触发描述
- 可选关键词
- 可选发送说明
- 资料文件或资料 URL

运行时逻辑：

1. 用户发消息。
2. 系统对自动发送资料规则做关键词匹配和语义匹配。
3. 命中规则后，将资料作为 assistant 消息的 outbound asset 发出。
4. 如果配置了发送说明，会自动附加到回复文本中。
5. 命中的规则会记录到消息元数据，前端聊天记录可见。
## 知识库如何触发

当前客服 Agent 只有勾选了知识库，消息才会触发该知识库检索。

现在的语义已经调整为：

- 勾选哪些知识库，就只用哪些知识库
- 一个都不勾选，就不启用知识库

不再有“未勾选等于全部知识库”的默认逻辑。

## 技能和知识库的勾选语义

客服 Agent 现在统一采用显式启用：

- 技能：不勾选就是不启用
- 知识库：不勾选就是不启用

这样配置更可控，也更符合客服场景。
## 模型怎么参与

这里涉及两类模型：

### 1. 对话模型

客服 Agent 绑定的 AI 模型负责最终回答。

它决定：

- 回复质量
- 工具调用能力
- 总结和表达能力

### 2. Embedding 模型

向量化和语义检索依赖 embedding 模型。

它决定：

- 知识库命中质量
- 自动发送资料规则的语义匹配能力

如果出现的问题是：

- 回答不自然
- 总结不好

优先调整对话模型。

如果出现的问题是：

- 文档明明有却检不出来
- 相近说法匹配不到手册或视频

优先检查 embedding 模型、分块策略和触发描述。
## 推荐配置方式

### 知识库适合放

- FAQ
- 产品说明
- 政策条款
- 长文档文本内容

### 自动发送资料适合放

- 操作手册 PDF
- 产品彩页
- 演示视频
- 安装包下载链接
- 售后表单模板

## 推荐实践

1. 把文本知识放知识库。
2. 把要发给用户的资料放自动发送资料规则。
3. 触发描述尽量写成用户真实说法，而不是内部标签。
4. 一条规则只发一类核心资料，避免一次命中过多附件。
5. 如果一个问题既要回答也要发文件，可以同时启用知识库和自动发送资料。
## 知识库的完整工作链路

知识库从上传到回答，大致经过 6 步：

1. 上传文档。支持 txt、docx 等可解析文本文件。
2. 文档解析。系统会把文档读取成纯文本。
3. 文本切块。长文会被拆成多个 chunk，避免一次检索过大。
4. 向量化。每个 chunk 都会用 embedding 模型转成向量。
5. 建索引。向量写入 Milvus；如果 Milvus 不可用，查询时会回退到数据库关键词匹配。
6. 对话检索。用户发消息时，系统先拿问题做检索，再把命中的片段拼进 system prompt，让对话模型生成最终回复。

所以知识库不是“把整篇文档原样发给用户”，而是“先找到相关片段，再交给 AI 整理成回答”。
## 什么时候会触发知识库

当前会满足这几个条件时触发：

1. 当前消息走的是绑定了客服 Agent 的会话。
2. 这个客服 Agent 勾选了至少一个知识库。
3. 用户的问题在语义上或关键词上能和知识片段匹配上。

如果一个客服 Agent 没有勾选知识库，即使系统里已经上传并向量化了很多文档，这次会话也不会使用它们。

这就是为什么“文档已经进了 Milvus”不等于“所有对话都会自动命中”。真正生效的是“当前 Agent 的绑定关系 + 当前用户问题的相关性”。
## 例子 1：产品参数问答

假设知识库里有一篇文档：

- 标题：X100 设备规格说明
- 内容：包含电池续航、充电时长、工作温度、接口类型

用户提问：

- “X100 能连续用多久？”

系统会做的事：

1. 拿这句话去检索。
2. 命中“电池续航 8 小时”这一段。
3. 把这一段放进 system prompt。
4. AI 输出类似“X100 在标准工况下可连续工作约 8 小时”。

这里用户收到的是“回答”，不是文档附件。
## 例子 2：售后政策问答

假设知识库里有一篇售后政策：

- “签收 7 天内可退，1 年质保，人为损坏除外”

用户提问：

- “你们这个支持退货吗？”

系统会检出退款条款相关 chunk，再由 AI 组织成更自然的话，比如：

- “支持退货，签收 7 天内可以申请；超过 7 天则按保修政策处理。”

如果没有知识库，AI 只能靠模型常识猜；有知识库，回答才会基于你自己的业务规则。
## 例子 3：知识库和自动发送资料一起工作

这是最常见的客服场景：

- 知识库里放《安装说明》的正文内容
- 自动发送资料规则里放《安装说明.pdf》

用户提问：

- “怎么安装？把安装文档也发我一下。”

系统可能同时发生两件事：

1. 知识库命中安装步骤，AI 先给出文字说明。
2. 自动发送资料规则命中“安装文档”，再把 PDF 附上。

最终用户既能看到 AI 的解释，也能收到文件。

这就是为什么知识库和自动发送资料最好分开配置。前者负责“答”，后者负责“发”。
## 什么时候需要换模型

先分清楚是换哪一类模型：

### 对话模型

负责最终回答。如果问题是：

- 回答太生硬
- 不会总结长上下文
- 工具调用能力弱

优先考虑换对话模型。

### Embedding 模型

负责知识库检索和自动资料规则语义匹配。如果问题是：

- 用户说法换了一种就搜不到
- 相似意思匹配不稳定
- 文档明明有但总命不中

优先考虑换 embedding 模型，或调整文档分块。

当前默认配置里，embedding 仍然走独立配置项，而不是跟聊天模型绑死。所以你完全可以保留现有对话模型，只单独调 embedding。
## 常见排查方法

如果你发现“文档已经上传，也已经向量化了，但回答没有用上知识库”，可以按这个顺序查：

1. 这个会话对应的客服 Agent 是否真的勾选了该知识库。
2. 文档导入状态是否成功，而不是 failed。
3. 文档内容是否真的包含用户问题对应的文本，而不只是文件名相关。
4. 用户问法是否太短、太口语，导致检索特征不足。
5. Milvus 是否可用；如果不可用，系统会回退关键词查询，效果通常会比向量检索弱。
6. embedding 模型是否合适，尤其是中英文混合场景。

一个很实用的经验是：

- 知识库里尽量放“可直接被引用的文本内容”
- 自动发送资料里放“需要发给用户的文件、视频、下载链接”

这样拆分后，命中路径会清晰很多。

## 当前限制与规划改进

以下为知识库/RAG 在代码中的**现状**与**路线图**（完整产品规划见 [roadmap.zh.md](roadmap.zh.md)）。

### 现状（已实现）

| 项目 | 行为 |
|------|------|
| 分块 | 知识库级 `fixed` / `paragraph` / `markdown` / `semantic`，可配 size / overlap / min_chunk / semantic_threshold |
| Embedding | 知识库可覆盖 provider / model / api_base / dimension；全局 `.env` 为默认；本地上传 zip / Ollama |
| 检索 | `vector` / `keyword` / `hybrid`；hybrid 使用 RRF 融合向量与 BM25 关键词结果 |
| Rerank | `lexical`（内置）/ `cohere` / `bge` / `cross-encoder`，可配 provider、model、top_n |
| Query Rewriting | LLM 改写多视角查询（可开关），多查询结果 RRF 合并 |
| Parent-Child | semantic 策略下子块检索命中后替换为父块上下文 |
| 存储 | `document_chunks` 表持久化分块（含 `parent_content`）；Milvus 存向量 |
| 重嵌入 | 知识库级 `POST .../reindex`，含 `reindex_status` 进度与 `indexed_embedding_key` 指纹 |

管理后台：知识库 → **RAG 设置**。API：`PATCH /api/knowledge/bases/{id}`。

### 本地上传 Embedding 模型

无需 API，可将 [sentence-transformers](https://www.sbert.net/) / HuggingFace 模型目录打成 **zip** 上传：

1. 管理后台 **知识库** 页顶部 → **上传模型 zip**
2. 校验通过后，在 **RAG 设置** 中将提供商选为 **本地上传模型**，并选择对应模型
3. 将 **向量维度** 设为模型输出维度（选用模型时会自动填入）
4. 执行 **全量重嵌入**

依赖（后端需安装）：

```bash
pip install sentence-transformers
```

API：

- `GET /api/knowledge/embedding-models` — 列表
- `POST /api/knowledge/embedding-models/upload` — 上传 zip
- `DELETE /api/knowledge/embedding-models/{id}` — 删除

另支持 **Ollama** 本地推理（`embedding_provider=ollama`，无需上传，需本机运行 Ollama）。

### 全量重嵌入

切换 Embedding（或可选地变更分块策略）后：

1. 保存 RAG 设置
2. 在 **RAG 设置** 底部点击 **全量重嵌入**
3. 可选勾选 **同时按当前策略重新分块**

API：`POST /api/knowledge/bases/{id}/reindex`，body：`{ "rechunk": true }`。

系统会记录 `indexed_embedding_key`；配置与指纹不一致时，列表显示 **待重嵌入**。
